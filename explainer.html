What Is It

Two Screens, One Experience

The project has two components that work together in real time.

📺 The Display Board

Opened on a TV or any browser. Shows a 6-row × 22-column grid of animated split-flap tiles. Automatically plays a demo loop until a phone connects. The URL is /board.

  • Shows a QR code for phone pairing
  • Runs demo messages when idle
  • Animates every character change mechanically
  • Plays a real split-flap clacking sound
  • Shows live count of active boards worldwide

📱 The Mobile Controller

Opened on a phone by scanning the QR code. Lets you type messages and send them to the board. The URL is /controller?code=XXXXXX.

  • Live preview mirrors the board exactly
  • Up to 10 messages, loops automatically
  • Clock mode (live time display)
  • Color tiles (7 colors, paint cells like pixels)
  • Artwork mode with 5 preset designs
💡 No accounts, no app install, no database required. The TV generates a 6-character code. You scan the QR, your phone connects instantly. Sessions live in memory and expire after 24 hours of inactivity.
Technology

Tech Stack

The entire system uses no frontend framework — no React, no Vue, no Angular. Just vanilla HTML, CSS, and JavaScript on the front. Node.js on the back.

Backend

Node.js + Express

Serves the HTML pages and handles HTTP requests. Also runs the WebSocket server.

ws (WebSocket library)

Low-level WebSocket server. The TV and phone connect here and stay connected.

Supabase (optional)

Stores session data so boards survive server restarts. The app works without it — sessions just reset.

Frontend

Vanilla JS (ES Modules)

3 HTML pages: homepage, board, controller. All JS is written with zero dependencies.

CSS 3D Transforms

The mechanical flip animation is pure CSS — no canvas, no WebGL. Uses rotateX perspective for the fold effect.

Doto (Google Fonts)

A dot-matrix font that gives the display its authentic retro aesthetic.

Tooling

Jest (Unit Tests)

Tests for the spool engine, session manager, and WebSocket client.

Puppeteer (Screenshots)

Automated headless browser screenshots for visual QA. npm run screenshot saves PNGs for review.

Sentry + PostHog (optional)

Error tracking and analytics — injected at build time, both optional.

Why no framework?

The project prioritizes zero dependencies on the frontend for three reasons:

  1. Performance: No React bundle means faster load time on TVs and older phones.
  2. Control: The tile animation system needs precise timing control that frameworks often abstract away.
  3. Simplicity: The app has 3 pages with specific responsibilities. A framework would add complexity without value.

Key Architectural Decisions

Shared modules — spool.js and wsClient.js are used by both the board and controller pages. The character rotation logic runs identically on both sides, enabling the controller's live preview to mirror the board exactly.

Server as router only — The server never stores the display content long-term. It forwards phone messages to the TV. Supabase persistence is optional and fire-and-forget.

Base href tags — Both HTML files use <base href="/board/"> so relative paths resolve correctly when served from the URL paths /board and /controller.

0.0.0.0 binding — The server explicitly binds to all network interfaces so phones on the same LAN can connect. Without this, Windows only accepts localhost.

Core Mechanics

The Spool Engine

The most important file in the project is public/shared/spool.js (85 lines). It defines how every character transition works — and it enforces the fundamental rule that makes this feel mechanical.

🔑 The Golden Rule: To go from A to D, the tile must pass through B then C. There are no shortcuts. This is what makes the display feel like real hardware.

The SPOOL Array

Every tile cycles through this exact 65-character sequence. It never goes backwards.

const SPOOL = ` ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789!@#$()-+=;:'"%,.?/°roygbpw`;
// Position 0 = space  →  A=1, B=2 ... Z=26  →  0=27 ... 9=36  →  symbols ...  →  colors at end

How the Engine Works

Each tile tracks 3 things:

const tile = {
  current: 'A',   // what's showing now
  target:  'D',   // where we want to get
  stepsLeft: 3  // how many flips remain
}

Each animation frame, stepAll() advances every tile that hasn't settled yet — one step per tile per frame. It returns whether anything changed, and whether all tiles finished.

Distance Calculation

To go from one character to another:

stepsToReach('A', 'D')
// A is index 1, D is index 4
// 4 - 1 = 3 steps forward

stepsToReach('Z', 'A')
// Z is index 26, A is index 1
// 65 - 26 + 1 = 40 steps (wrap)

The spool always wraps forward. The longer the distance, the longer the tile flips before settling.

Interactive Demo — Type a Word

Watch the tiles cycle through every intermediate character on their way to your word.

Each tile flips one step per frame. Long distances = more flips. Notice how all tiles start simultaneously.

Color Characters

The last 7 positions in the SPOOL are color tiles: r o y g b p w. They're stored as lowercase to avoid confusion with letters. When rendered, they show as solid color blocks instead of text characters.

Visual Engineering

The Flip Animation

Each tile uses a 4-panel CSS 3D architecture. This is the most complex part of the frontend — it simulates the exact visual behaviour of a real mechanical split-flap tile.

How a Real Flap Works

In a real split-flap display, each tile is a card that spins on an axis at its center. The top half shows the current character; the bottom half is fixed from the previous character. As it flips, the top folds down and the new bottom swings up from behind.

The 4-Panel Structure

Every tile in the DOM contains 4 overlapping elements, each with a different z-index and role:

z:1 .top-half-static Resting top — shows current char. Updated after flip.
z:2 .bottom-flap-animating Incoming char top. Pre-rotated to –90° (edge-on). Unfolds to 0° during flip.
z:3 .top-flap-animating Current char top. Folds from 0° → 90° (folds down) during flip.
— .bottom-half-static Incoming char bottom. Snapped instantly. Never animates.

The JS Orchestration Loop

function animLoop(timestamp) {
  // Throttle to ~60ms intervals (flip speed)
  if (timestamp - lastStep >= 60) {
    const { anyChanged, allSettled } = stepAll(grid);

    // For each tile that changed this step, trigger CSS flip
    grid.forEach((tile, i) => {
      if (tile.justChanged) {
        triggerFlip(tileEls[i], tile.current); // adds .flipping class
        pendingFlips++;
      }
    });

    if (allSettled) { animRunning = false; return; }
  }
  requestAnimationFrame(animLoop);
}

tileEl.addEventListener('animationend', () => {
  // Copy new char to static panels, remove .flipping
  pendingFlips--;
  if (pendingFlips === 0 && !animRunning) stopAudio();
});
🎨 Surface Texture: ::before pseudo-elements on each tile half apply a scanline grain + angled light-sheen at z-index: 4 (above all flip panels). This makes characters look printed into the tile material. Three fold-score lines in the lower half simulate the physical crease marks on real flap cards.
Sound Design

Integrating the Flip Sound

The mechanical clacking sound is what makes the experience feel real. The audio implementation has several non-obvious constraints to solve.

The Problem with <audio loop>

The obvious approach — an HTML <audio> element with loop — has a gap between loop cycles. You can hear a tiny silence every few seconds. For a continuous mechanical clacking sound, this is completely wrong.

Also: autoplay policies on modern browsers block audio until a user gesture occurs.

The Solution: Web Audio API

We use AudioBufferSourceNode with loop = true. This loops with zero gap — it's a seamless buffer that cycles perfectly.

The raw audio bytes are fetched on page load (no user gesture needed for fetching). The AudioContext is only created on first user interaction — satisfying the browser policy.

The Audio Lifecycle

1
Page loads → fetch('/assets/audio/split-flap.wav') → store rawAudio bytes
No user gesture needed to fetch. The bytes are held in memory, not decoded yet.
2
User clicks "Skip" → audioCtx = new AudioContext()
AudioContext requires a gesture. We use the Skip button as the unlock moment.
3
audioCtx.decodeAudioData(rawAudio.slice(0)) → audioBuffer
Decoding happens once, async. If the board is already animating, startAudio() fires immediately after decode.
4
First tile starts flipping → pendingFlips++ → startAudio()
Audio starts playing. The loop runs continuously.
5
animationend on each tile → pendingFlips--
When pendingFlips reaches 0 AND the animation loop is not running: stopAudio().
// pendingFlips counter — ties audio exactly to visual animation
function triggerFlip(tileEl, newChar) {
  tileEl.classList.add('flipping');
  pendingFlips++;
  if (!audioPlaying) startAudio();

  tileEl.addEventListener('animationend', () => {
    pendingFlips--;
    if (pendingFlips === 0 && !animRunning) stopAudio();
  }, { once: true });
}
Real-Time Communication

How the Controller Connects to the Board

The pairing system uses WebSockets — a persistent, two-way connection. Unlike regular HTTP requests (which close after each message), WebSockets stay open indefinitely. This is what enables real-time control with no lag.

The Pair Code System

When the TV opens, the server generates a 6-character code using only easy-to-read characters (no O, 0, I, 1 — characters that look similar). This code is embedded in a QR code on screen.

Valid alphabet: ABCDEFGHJKLMNPQRSTUVWXYZ23456789
Excluded: O, 0, I, 1 ← visually ambiguous characters

Full Pairing Sequence

📺
TV opens /board → connects to WebSocket → sends { type: 'tv_hello' }
TV may include a sessionId from localStorage to resume a previous session.
⚙
Server creates session → responds { type: 'tv_paired', sessionId, pairCode }
sessionId stored in TV's localStorage. pairCode shown as QR on screen.
📱
User scans QR → phone opens /controller?code=XXXXXX → sends { type: 'phone_hello', pairCode }
Server checks: is the TV's WebSocket open? If not → sends board_offline and closes. If yes → approves.
⚙
Server sends { type: 'phone_approved' } to phone AND { type: 'phone_approved' } to TV
TV hides QR screen, shows the board. Phone hides the connect screen, shows controller UI.
📱
User types → phone sends { type: 'phone_send', payload: { rows, mode } }
Server forwards to TV as { type: 'display_update', rows }. TV animates.

Phone → Server

// Join with a pair code

  payload: { rows: [...], mode: 'message' }}

{ type: 'phone_next' }

  pairCode: 'AB2C3D' }

// Tell both devices they're connected
{ type: 'phone_approved' }

// Forward content to TV
{ type: 'display_update', rows: [...] }

// Tell TV the phone disconnected

// Tell phone the TV isn't open
{ type: 'board_offline' }

// Live count for homepage
{ type: 'boards_live', count: 3 }

Network connections drop — especially on mobile. The wsClient.js wrapper handles this:

Automatic Reconnection

On disconnect, wsClient waits 500ms then tries to reconnect. Each failed attempt doubles the wait (500ms → 1s → 2s → 4s → max 30s). On success, it replays any buffered messages.

Safari Safari Safari

iOS Safari aggressively kills background WebSockets. The phone sends a ping message every 10 seconds to keep the connection alive. The server has a 20-second heartbeat check — sockets that don't respond get terminated.

Phone Disconnect Behavior

When the phone disconnects, the board stays visible (it doesn't return to the QR screen). It flips to DISCONNECTED. When the phone reconnects and re-approves, it flips back to DEVICE CONNECTED.

Visual Design

UI Design Iterations

The visual design evolved significantly over the project's lifetime, driven by authenticity goals and usability feedback.

Font Evolution

Phase 1: BlackLine / Custom

Early experiments with pixel-style fonts. The rendering was incorrect — tiles had dark characters on dark backgrounds, and sizing didn't match the split-flap aesthetic.

rendering issues

Phase 2: SplitFlapTV (Stencil)

A stencil-style font that needed a white backing layer behind each character to be visible. This added complexity and didn't render well at small sizes.

stencil workaround needed

Phase 3: Doto ✓

A standard dot-matrix Google Font. Clean white characters on dark panels. No tricks needed. Weight 700, ROND: 0 for square dots. Loads via Google Fonts CDN — fast and universal.

DOTO

Controller Input Evolution

v1: Textarea

Simple HTML textarea per row. Easy to implement but had problems: no per-character control, couldn't enforce 22-char limit visually, couldn't render color tiles inline, font didn't match the board.

v2: Character Grid (divs)

Replaced with a 22×6 CSS grid of individual <div> cells. A hidden off-screen <input> captures keyboard events — this is required for virtual keyboard on iOS/Android since inputs trigger the keyboard but divs don't.

v3: Drag-to-Swap ✓

Added 320ms long-press to enter drag mode, then pointer-move to drag color tiles or characters to other cells. Empty targets = move, filled targets = swap. This made color artwork editing practical.

Tile Visual Refinements

Aspect Ratio Change

Original tiles were roughly square. Real split-flap tiles are portrait orientation — taller than wide. The final ratio is 2.2:4.3 (width:height), which matches real airport/train station hardware.

Base CSS: width: 2.2rem; height: 4.3rem. On board pages, tiles become responsive: width: 100%; aspect-ratio: 2.2/4.3.

Surface Details

Added in iterations:

  • Center seam: 2px black border between top/bottom half
  • Seam glow: 1px light line just below the seam
  • Fold-score lines: 3 gradient creases in the lower half at 60%, 74.5%, 89%
  • Texture overlay: Scanline grain + angled light-sheen via CSS gradients

Engineering Challenges

Bug Log

The most interesting bugs and how they were solved. Each one reveals something about how browsers, mobile operating systems, or JavaScript actually behave in production.

BUG iOS Safari: phone stuck at "Waiting for approval" forever

Root cause: Safari on iOS has aggressive background tab suspension. When the user switches apps to scan the QR code with the camera, the WebSocket connection was being killed in the background. When the page regained focus, the socket was in a "zombie" state — appeared open but was not delivering messages.

Symptoms: The phone would send phone_hello, the server would approve, but the phone's onMessage handler never fired because the socket was stale.

FIX Three-part solution: (1) Server heartbeat — pings every 20s and closes sockets that don't respond. (2) Phone-side watchdog timer — if phone_approved doesn't arrive in 8 seconds, force-disconnect and reconnect. (3) Head-start connection — the WebSocket connects before fonts load, giving the handshake maximum time on slow connections.

BUG Safari: URL parameter lost after scan (?code=XXXXXX stripped)

Root cause: Safari sometimes "upgrades" bare IP:port URLs typed into the address bar to HTTPS, and in certain navigation flows was stripping the ?code= query parameter from the URL before the JS ran.

Symptoms: Phone opens the controller page, the URL shows no code, and the UI shows "no pair code found." The user is stuck.

FIX Two-layer fallback: (1) QR code always encodes the full http:// URL including the pair code. (2) If JS can't find the code in location.search, it falls back to the last code in localStorage, then prompts the user with a manual entry dialog. Cache-busting query params added to all JS imports to prevent Safari from serving stale modules.

BUG Socket fighting — infinite reconnection loop creating duplicate connections

Root cause: When a network blip caused the WebSocket to close, the reconnection logic would create a new connection while the old "zombie" socket was still technically in a closing state. The server would see two connections for the same session, routing messages to the wrong socket.

Symptoms: Board and phone lose sync. Messages would sometimes arrive, sometimes not. The controller showed "connected" but the board wasn't responding.

FIX Added a connection lock flag (_isConnecting) to the wsClient that prevents new connection attempts while one is in progress. Added a stale-close guard on the server: when a new TV/phone socket registers for a session, the server closes any existing socket for that slot first. The old socket's close event is ignored if it fires after replacement.

BUG Black screen on reconnect — board goes blank after network recovery

Root cause: When the phone reconnected after a drop, the controller sent phone_send with the current rows before the board's WebSocket had finished re-registering with the server. The message arrived but was routed to the old (null) socket entry, silently dropped.

Symptoms: After reconnecting, the board tiles would all flip to blank (space characters) because the board re-initialized but received no content.

FIX The server stores currentRows in the session. When the TV reconnects, the server immediately re-sends the last known rows via display_update. The controller's reconnect handler also re-sends on successful phone_approved confirmation.

BUG Audio double-click bug — had to click Skip twice to hear sound

Root cause: The audio buffer fetch and decodeAudioData are async. On the first Skip click, the AudioContext was created and decode started — but the board was already animating. startAudio() was called during decode, but the buffer wasn't ready yet, so it silently did nothing. Sound only started on the next flip.

Symptoms: Users had to click Skip twice to hear the clacking sound. First click started animation, second click (triggering another flip) finally got audio going.

FIX After decodeAudioData resolves, immediately check if animRunning is true — if so, call startAudio() right then. This bridges the gap between "user unlocked audio" and "board is already flipping."

SECURITY XSS via unescaped environment variables injected into HTML

Root cause: The server injects Sentry DSN and PostHog API keys into the HTML at startup time. These came directly from environment variables without sanitization. A malformed env var could close the script tag and inject arbitrary JS.

Symptoms: Theoretical — but a critical vulnerability if exploited in a misconfigured deployment environment.

FIX All injected values are now run through an escape function that replaces <, >, ", ', and / with their HTML entity equivalents before injection. Committed as a dedicated security fix PR.

SECURITY Pair codes not truly random — Math.random() used instead of crypto API

Root cause: The original pair code generator used Math.random(), which is not cryptographically secure. In theory, if an attacker could predict the seed, they could predict upcoming pair codes and hijack sessions.

FIX Replaced with crypto.getRandomValues() (Web Crypto API), which uses the OS's cryptographically secure random number generator. This ensures pair codes cannot be predicted or enumerated.

PERF O(N) session lookup — server slows with many active sessions

Root cause: The original session manager stored sessions in an array. Finding a session by pair code required looping through every session — O(N) performance. Similarly, counting active sessions looped through all sessions.

FIX Switched to two Maps: one keyed by session ID, one keyed by pair code. Lookups are now O(1). Active session count maintained as a separate integer (incremented/decremented on state changes) rather than recalculated each time.

Everything except PORT is optional. Without Supabase, sessions reset on server restart.

0.0.0.0 Binding (Critical)

The server must bind to '0.0.0.0' explicitly: server.listen(PORT, '0.0.0.0'). Without this, Node.js on Windows binds to IPv6 (::) only and rejects phone connections from the local network.

QR Code IP Detection

getLanIp() collects all non-internal IPv4 addresses and prefers 192.168.x.x then 10.x.x.x. This avoids returning virtual adapter IPs from WSL, VMware, or Hyper-V which wouldn't be reachable by phones.

http:// in QR Code

The QR URL always has an explicit http:// prefix. Safari will auto-upgrade bare IP:port URLs to HTTPS, which fails with a self-hosted server. The full URL prevents this.

Supabase Session Persistence

Without Supabase, the server stores all sessions in JavaScript memory. If the server restarts, all active sessions are lost — boards would show the QR screen again. Supabase is a PostgreSQL-backed cloud service that the server uses as a fire-and-forget persistence layer:
